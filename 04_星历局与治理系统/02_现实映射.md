## 1. 文档目标

本文件回答的问题是：

> 星历局在现实技术中究竟对应什么？

修订后的答案不再是“一个大服务器集合”或“一个带美学包装的 STUN/TURN 中心”，而是：

> 一套由**公共索引、见证记录、广播镜像、裁决存证、治理执行**组成的协议化公共网络。

它映射的现实对象包括：
- 公共注册服务；
- 分布式索引层；
- 广播镜像网络；
- 审计日志与争议存证；
- 治理资格与执行控制层。

---

## 2. 总体映射框架

```text
世界观：星历局
→ 现实：公共基础设施协议网络

世界观：见证者
→ 现实：注册服务 / 验证节点 / 审计节点

世界观：宇宙档案
→ 现实：append-only log / arbitration record / public metadata store

世界观：制裁执行
→ 现实：公共服务拒绝、权限冻结、公共索引状态更新
```

---

## 3. 星历局五类节点的现实映射

| 世界观节点 | 现实映射 | 功能边界 |
|-----------|---------|---------|
| 注册见证节点 | Registration service / witness cluster | 记录公开身份声明与状态变更 |
| 发现索引节点 | discovery index / DHT overlay / public service directory | 提供公共查找与索引 |
| 广播镜像节点 | broadcast mirror / pub-sub relay / event replication node | 镜像公共公告与警报 |
| 仲裁记录节点 | append-only dispute log / arbitration DB / tamper-evident archive | 记录裁决、申诉、继承与争议 |
| 治理执行见证节点 | governance executor + audit witness | 记录并执行已通过的公共后果 |

关键点在于：
- 这些节点可以协作，但不应全部集中在一个单体后端；
- 其核心是**公共秩序服务**，不是平台全能控制台。

---

## 4. 星历局不应被错误映射成什么

## 4.1 它不只是 STUN/TURN
STUN/TURN 只解决连接建立与中继问题，不能解释：
- 身份见证；
- 公共争议记录；
- 广播镜像；
- 治理执行；
- 制裁状态写入。

## 4.2 它不应等同单一云平台
如果将星历局现实化为某个单一云厂商上的中心服务，那么世界观中的：
- 去中心化演化；
- 共识冗余；
- 审计透明；
- 分裂恢复；

都会沦为空话。

## 4.3 它不等于传统国家政府
星历局更像：
- DNS + 公共日志；
- package registry + 审计层；
- 社区治理网络 + 执行网关；
- 多节点公共基础设施联盟。

---

## 5. 公共索引层映射

### 世界观表达
星图、注册、发现、公共目录。

### 现实对应
- service registry；
- distributed directory；
- DHT + cache；
- public searchable metadata。

### 设计边界
公共索引层只处理：
- 可公开的数据；
- 可验证的身份引用；
- 可查找的服务与状态。

它不应：
- 保存私密虫洞内容；
- 代替节点本地数据库；
- 冒充主权核心源。

---

## 6. 仲裁与记录层映射

### 世界观表达
宇宙档案、裁决留痕、继承争议、制裁记录。

### 现实对应
- append-only log；
- tamper-evident event ledger；
- arbitration record store；
- signed evidence registry。

### 必须记录的内容
- 提案发布与通过；
- 投票摘要与资格状态；
- 继承、恢复、申诉事件；
- 制裁生效与解除；
- 星历局节点本身的操作日志。

### 不应记录的内容
- 私密消息正文；
- 本地私有数据；
- 无法公开披露的密钥材料。

---

## 7. 广播镜像层映射

### 世界观表达
全宇宙公告、安全警报、治理传播、公共警示。

### 现实对应
- pub/sub 系统；
- replicated event feed；
- WebSocket / SSE / queue fanout；
- gossip-assisted broadcast mirror。

### 现实边界
广播镜像层不是内容审查机器，而是公共信息传播层。它需要的不是无限审查权，而是：
- 优先级；
- 范围控制；
- 限流；
- 反污染策略；
- 来源签名验证。

---

## 8. 治理系统映射

## 8.1 世界观表达
提案、表决、执行、上诉、制裁。

## 8.2 现实对应
- proposal registry；
- eligibility checking；
- voting engine；
- execution queue；
- appeal workflow。

## 8.3 关键修正
治理不能再被现实化为“简单投票箱”。

它至少需要五层：
1. 提案资格检查；
2. 身份与状态检查；
3. 双院或多维票权计算；
4. 执行窗口；
5. 申诉与回滚记录。

---

## 9. 双院制的现实映射

### 世界观表达
为了防止廉价节点治理劫持，公共治理采用双院制。

### 现实对应
- **星球院**：按合格主体数计票，保护基础平等；
- **贡献院**：按长期公共贡献、基础设施维护、治理信誉加权。

### 为什么需要双院
如果只有“一个节点一票”：
- 女巫攻击成本极低；
- 假星球批量制造可操纵公共规则；
- 小型伪装集群可绑架公共广播与索引层。

如果只有“贡献加权”：
- 资源寡头可能永久统治；
- 早期强者会锁死后续进入者。

所以双院制的现实意义，是在：
- 平等；
- 持续贡献；
- 反女巫；
- 反寡头；

之间取得制度平衡。

## 10. 制裁的现实映射

### 世界观表达
共同体可以对违规主体执行公共限制。

### 现实对应
不是物理征服，而是：
- 从公共索引中标记风险；
- 关闭其高级广播资格；
- 暂停其治理票权；
- 暂停公共市场的某些高风险权限；
- 限制其使用公共中继与公共见证。

### 关键边界
这类制裁影响的是**公共权利**，不是本地主权本身。

现实上相当于：
- 拒绝其使用公共 API；
- 将其标记为高风险 subject；
- 拒绝其参与部分共享网络层。

而不是：
- 删除其服务器；
- 擅自接管其本地数据；
- 远程抹除其私钥。

## 11. 星历局分裂与恢复的现实映射

### 世界观表达
星历局可能因治理冲突、节点失效、路线分歧而分裂。

### 现实对应
- registry fork；
- public log divergence；
- governance network partition；
- mirror inconsistency。

### 恢复要求
现实系统必须支持：
- 多镜像冗余；
- 签名校验；
- 日志回放；
- 冲突合并；
- 分区恢复后的状态重建。

### 世界观意义
星历局必须可分裂后恢复，才能证明自己不是靠单一点神权维持秩序，而是靠协议与共同体继续存在。

## 12. 技术架构建议图

```text
Public Registry Layer
- registration witnesses
- identity/public-state records

Discovery Layer
- service directory
- DHT/cache index
- public map metadata

Broadcast Layer
- mirrored feed
- priority event pub/sub
- anti-spam fanout control

Arbitration Record Layer
- append-only logs
- dispute and sanction records
- signed evidence references

Governance Execution Layer
- proposal workflow
- dual-chamber tally
- execution witness and appeal state
```

## 13. 与其他系统的现实接口

## 13.1 与身份系统
接收：
- 公共身份；
- 状态标记；
- 恢复、继承、争议信号。

## 13.2 与连接系统
提供：
- 公共发现目录；
- 中继资格状态；
- 风险标记。

## 13.3 与经济系统
提供：
- 市场准入状态；
- 制裁状态；
- 裁决记录查询。

## 13.4 与声誉系统
交换：
- 公共违规事件；
- 长期基础设施贡献；
- 治理参与与滥用记录。

## 14. 完成标准

如果本文件映射成功，应能清晰回答：

1. 为什么星历局不应只被映射成单一中心服务器；
2. 为什么治理不能简化成投票计数器；
3. 为什么制裁影响的是公共权利而不是本地主权；
4. 为什么广播、裁决、发现与注册必须视为不同层级的公共基础设施；
5. 为什么星历局必须可镜像、可审计、可恢复。
