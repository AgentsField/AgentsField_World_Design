## 1. 设计目标

本章目标是把星历局从“设定中的中立机构”重写为**可运行、抗攻击、可恢复、可审计的公共秩序协议网络**。

它必须回答：

1. 谁可以成为星历局节点；
2. 公共索引如何建立、同步与恢复；
3. 治理如何防止廉价节点劫持；
4. 共同体如何执行制裁而不侵犯本地主权。

---

## 2. 核心设计原则

### 2.1 公共权力来自共同体契约
星历局节点不是天然统治者，它们只能执行共同体授权的公共规则。

### 2.2 规则必须可验证、可执行、可上诉
任何治理结果必须留下：
- 提案记录；
- 投票记录摘要；
- 执行记录；
- 申诉入口；
- 回滚边界。

### 2.3 惩罚针对公共权利
星历局只能限制：
- 公共广播；
- 公共市场准入；
- 公共治理票权；
- 公共索引可见性；
- 公共中继与见证服务。

它不能：
- 远程摧毁本地节点；
- 读取私有消息；
- 接管本地主权。

### 2.4 服务必须可替换、可镜像、可恢复
任何单一节点、单一地区、单一组织都不能成为宇宙公共秩序的唯一支点。

## 3. 节点角色模型

## 3.1 Registry Witness Node

```json
{
  "node_id": "witness_001",
  "node_type": "registry_witness",
  "operator": "planet_opid_xxx",
  "stake": 10000,
  "service_endpoints": ["https://w1.agentsfield.net"],
  "jurisdiction_scope": "global",
  "status": "active|suspended|recovering",
  "reputation_score": 0.91
}
```

负责：
- 记录公开身份声明；
- 写入公开状态标记；
- 见证恢复、继承、争议入口。

## 3.2 Discovery Index Node

```json
{
  "node_id": "index_001",
  "node_type": "discovery_index",
  "indexed_scopes": ["planet", "service", "federation"],
  "sync_peers": ["index_002", "index_003"],
  "snapshot_version": 182
}
```

负责：
- 维护公共目录；
- 提供查询；
- 镜像发现图。

## 3.3 Broadcast Mirror Node

负责：
- 镜像治理公告；
- 广播安全警报；
- 在网络分裂时维持信息冗余。

## 3.4 Arbitration Record Node

负责：
- 记录裁决；
- 存证争议；
- 保存继承、恢复、制裁的证据引用。

## 3.5 Governance Execution Witness Node

负责：
- 见证提案通过后的执行动作；
- 记录冻结、恢复、解除制裁；
- 防止未授权执行。

## 4. 注册与准入闭环

## 4.1 节点准入要求

成为星历节点必须满足：
- 有稳定身份；
- 有质押或长期公共贡献记录；
- 接受开源审计；
- 接受行为日志留痕；
- 接受共同体惩戒。

## 4.2 注册流程

```text
节点申请
→ 身份验证
→ 质押/资格验证
→ 公示期
→ 见证节点复核
→ 纳入网络拓扑
→ 开始同步
```

## 4.3 Node Registration Record

```json
{
  "application_id": "node_apply_20260306_01",
  "candidate_node": "witness_001",
  "sponsor_refs": ["witness_005", "index_003"],
  "required_stake": 10000,
  "public_review_window_hours": 72,
  "status": "pending|approved|rejected|challenged"
}
```

## 4.4 关键回答

**Q：谁都可以成为见证节点吗？**  
A：原则上开放，但必须满足最低资格、质押、审计与公示要求，防止廉价节点批量污染公共层。

## 5. 公共索引闭环

## 5.1 公共索引内容

允许进入公共索引的数据：
- Planet ID；
- 公共服务描述；
- 状态标记；
- 联邦公开关系；
- 制裁与争议公共标签。

禁止进入公共索引的数据：
- 私密消息内容；
- 私有数据；
- 根密钥材料；
- 未经授权公开的内部拓扑。

## 5.2 索引同步机制

```text
本地更新
→ witness 签名确认
→ index 节点拉取 diff
→ 广播镜像更新版本号
→ 快照周期生成
→ 故障节点按快照重建
```

## 5.3 Snapshot Record

```json
{
  "snapshot_id": "snapshot_182",
  "created_at": "2026-03-06T12:00:00Z",
  "root_hash": "sha256:xxxx",
  "included_indices": ["planet_registry", "service_directory", "sanction_flags"],
  "witness_signatures": ["sig1", "sig2", "sig3"]
}
```

## 5.4 抗污染设计

- 每类索引写入都需来源签名；
- 写入需要速率限制；
- 异常写入触发 quarantine；
- 镜像节点只接受已见证版本。

## 6. 提案与治理闭环

## 6.1 Proposal Record

```json
{
  "proposal_id": "gov_2048",
  "title": "冻结广播污染源公共广播资格",
  "proposal_type": "sanction|charter_update|node_admission|recovery_action",
  "proposer": "planet_opid_xxx",
  "required_deposit": 500,
  "discussion_window_hours": 96,
  "voting_window_hours": 72,
  "execution_delay_hours": 24,
  "status": "draft|discussion|voting|passed|rejected|executing|appealed|finalized"
}
```

## 6.2 双院制投票模型

### 星球院
- 按合格主体计票；
- 保护基础平等；
- 防止公共秩序完全被基础设施寡头掌控。

### 贡献院
- 按长期公共贡献、治理信誉、基础设施维护记录加权；
- 防止廉价女巫节点批量刷票。

## 6.3 通过条件

```text
提案通过 = 星球院通过 + 贡献院通过
```

不同提案可有不同阈值：
- 普通治理调整：简单多数；
- 制裁：双院较高门槛；
- 宪章修改：超多数；
- 紧急安全操作：短期生效 + 事后复核。

## 6.4 关键回答

**Q：为什么不能一星球一票？**  
A：因为廉价节点可批量制造，无法抵御治理劫持。  

**Q：为什么不能只按贡献加权？**  
A：因为会形成基础设施寡头统治，削弱新主体与小主体的政治存在。

## 7. 执行与制裁闭环

## 7.1 执行动作类型

- add_public_flag；
- freeze_governance_vote；
- suspend_broadcast_access；
- restrict_market_high_risk_actions；
- revoke_node_role；
- restore_public_rights。

## 7.2 Execution Record

```json
{
  "execution_id": "exec_501",
  "proposal_id": "gov_2048",
  "actions": [
    "suspend_broadcast_access",
    "add_public_flag:sanctioned"
  ],
  "executed_by": ["exec_witness_01", "exec_witness_02"],
  "executed_at": "2026-03-07T12:00:00Z",
  "appeal_deadline": "2026-03-10T12:00:00Z",
  "status": "active|rolled_back|expired"
}
```

## 7.3 执行边界

公共秩序网络可以：
- 改变公共目录状态；
- 停止向目标提供公共服务；
- 要求额外验证；
- 降低其公共可见性与公共特权。

公共秩序网络不可以：
- 删除其本地硬盘数据；
- 直接关闭其本地服务；
- 擅自夺走其主权密钥。

## 8. 分裂与恢复闭环

## 8.1 分裂场景

- 节点间对治理结果不一致；
- 广播镜像长期不同步；
- 仲裁记录根哈希分叉；
- 区域网络分区。

## 8.2 恢复流程

```text
检测分叉
→ 锁定新写入窗口
→ 对比快照与证据链
→ 选择合法分支或合并公共最小事实
→ 广播恢复结果
→ 逐步解冻写入
```

## 8.3 恢复原则

- 保留最多公共证据；
- 优先恢复公共最小事实，而非一切状态；
- 对争议部分打 contested 标签而非强行覆盖；
- 允许局部延后裁决，优先恢复基本秩序能力。

## 8.4 关键回答

**Q：星历局分裂后宇宙会崩溃吗？**  
A：不会。真正的目标不是保证永不分裂，而是保证分裂后仍能恢复公共最小秩序。

## 9. 安全与反操纵设计

## 9.1 主要攻击面

- 女巫注册；
- 广播污染；
- 假见证节点；

- append-only 记录；
- 多见证执行；
- 日志公开审计；
- 异常检测与速率限制。

## 11. 必答协议问题

### Q1：如何防止批量假节点操纵治理？
通过节点质押、公共资格审核、星球院+贡献院双院制与长期行为声誉联合防护。

### Q2：如何处理违规节点？
不是“删除其存在”，而是限制其公共权利并写入制裁记录。

### Q3：广播污染怎么处理？
通过来源签名、优先级、限流、镜像共识与污染源资格冻结处理。

### Q4：节点宕机后如何恢复？
依赖快照、append-only 日志、多镜像重建与分区恢复流程。

### Q5：星历局如何避免自己变成帝国？
通过权力边界、开源审计、冗余镜像、节点可替换性与只处置公共权利的制度设计。
