## 🧱 设计目标

开发设计需要把声誉与成就从单值积分机制升级为：
- 多维声誉画像；
- 可追溯的成就授予与褫夺系统；
- 可保留的污点记录；
- 可修订的历史注记；
- 与经济、治理、外交、建筑系统联动的信任层。

## 二、声誉维度计算建议

### 1. 维度说明
- `fulfillment`：基于交易、交付、稳定性
- `public_service`：基于公共设施、救援、开放贡献
- `governance`：基于提案、守法、裁决合作度
- `exploration`：基于发现、探测、开拓
- `defense`：基于防护、守卫、应急支持
- `stain`：负向记录累积程度

### 2. 计算原则
- 正向维度可缓慢积累；
- 污点维度不能被简单用时间完全抹平；
- 同一事件可同时影响多个维度；
- 不同联邦可对权重有局部调整，但需保留统一基础层。

### 示例伪代码

```python
def apply_event(profile, event):
    if event.type == "fulfilled_order":
        profile["fulfillment"] += 2
    if event.type == "public_rescue":
        profile["public_service"] += 5
        profile["defense"] += 2
    if event.type == "broadcast_pollution_confirmed":
        profile["stain"] += 12
        profile["governance"] -= 4
    return clamp_profile(profile)
```

## 三、成就授予逻辑

### 授予条件的推荐结构
每项成就建议包含：
- `trigger_rule`
- `required_evidence_count`
- `required_witness_types`
- `granting_authority`
- `revocable`（是否可褫夺）

### 示例：永灯守望者

```json
{
  "type": "public_infrastructure",
  "trigger_rule": {
    "public_service_uptime_days": 365,
    "critical_incidents": 0
  },
  "required_evidence_count": 3,
  "required_witness_types": ["registry", "community", "audit"],
  "granting_authority": "federation_council",
  "revocable": true
}
```

### 授予流程
1. 条件触发；
2. 自动或人工收集证据；
3. 治理或授权机构审核；
4. 公示期；
5. 正式授予并写入编年。

## 四、污点记录与修复逻辑

### 污点写入条件
- 必须有正式裁决或同级别可验证结论；
- 必须记录影响范围；
- 必须附上最少一条恢复路径说明。

### 修复机制
可通过以下方式降低负面影响：
- 完成赔付；
- 长期无新的同类违规；
- 提供公共修复服务；
- 接受并完成治理要求的整改。

### 注意
- `stain` 可下降，但 `StainRecord` 历史条目不应删除；
- 前端需展示“已修复污点”与“未修复污点”区别。

## 五、历史修订与翻案机制

### 开启条件
- 新证据；
- 程序缺陷暴露；
- 旧裁决被证伪；
- 跨联邦证据合并。

### 修订结果
- 维持原判；
- 附加注释；
- 调整维度分值；
- 成就状态改为 `disputed`；
- 完全褫夺；
- 重授或平反。

### UI 状态建议
- `active`
- `disputed`
- `revoked`
- `restored`
- `annotated`

## 六、关键接口设计

### 1. 查询声誉画像
`GET /api/v1/reputation/{opid}`

### 2. 查询成就列表
`GET /api/v1/achievements/{opid}`

### 3. 提交成就授予审核
`POST /api/v1/achievements/review`

### 4. 写入污点记录
`POST /api/v1/stains/create`

### 5. 提交翻案申请
`POST /api/v1/history/revision`

### 6. 查询历史修订记录
`GET /api/v1/history/revisions/{target_id}`

## 八、测试要点

### 功能测试
- 单一事件是否正确影响多个维度；
- 成就授予条件是否正确触发；
- 污点修复后是否仅调整影响而非删除历史；
- 翻案后成就状态与声誉值是否同步更新。

### 压力测试
- 高频订单环境下履约声誉实时更新能力；
- 大规模历史回填时的声誉重算性能；
- 多联邦同时提交翻案申请时的一致性。

### 安全测试
- 伪造授勋证据；
- 未经裁决写入污点；
- 恶意刷取公共声誉；
- 利用别名身份逃逸历史污点。
