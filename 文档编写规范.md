## 📝 文档规范

### 六类文档职责划分

| 文档类型 | 目标读者 | 核心职责 | 输出物 | 禁止内容 |
|---------|---------|---------|--------|---------|
| 00_README.md | 全体成员 | 章节总览与导航 | 概述、进度、依赖关系 | 详细实现内容 |
| 01_世界观设定 | 全体成员 | 故事化描述系统 | 术语表、用户体验 | 技术细节、代码 |
| 02_现实映射 | 产品+技术 | 建立技术与世界观映射 | 映射表、转换逻辑 | 纯技术实现 |
| 03_资源与美术 | 设计师 | 明确视觉资源需求 | 资源清单、优先级 | 技术实现 |
| 04_开发设计 | 工程师 | 提供技术实现方案 | API、数据结构、代码 | 世界观故事 |
| 05_事件与演化 | 运营策划 | 设计动态内容 | 事件表、演化路径 | 底层技术 |

### 文件命名

- 章节文件夹：`XX_章节名称/`（XX为两位数字）
- 章节目录文件：`README.md`（每个章节文件夹下必备）
- 子文档文件：`0X_部分名称.md`（X为1-5）


### 章节文件结构

```
XX_章节名称/
├── README.md              # 章节总览（必备）
├── 01_世界观设定.md
├── 02_现实映射.md
├── 03_资源与美术.md
├── 04_开发设计.md
└── 05_事件与演化.md
```

### 编写原则

1. **章节目录先行**：每个章节先创建 README.md 作为总览
2. **世界观优先**：先写世界观（01），再写映射（02），最后写技术（04）
3. **保持一致**：术语和概念在所有文档中统一（参考 GLOSSARY.md）
4. **可视化优先**：多用表格、流程图、示例代码
5. **迭代更新**：每个文档都有更新记录和待定问题清单
6. **职责单一**：每个文档只做自己职责范围内的事
7. **相互引用**：文档间通过相对路径引用，保持关联

### 质量标准

每个文档完成后需满足：

- ✅ **完整性**：覆盖章节所需的所有核心内容
- ✅ **准确性**：技术映射和术语定义无歧义
- ✅ **可读性**：非目标读者也能看懂70%内容
- ✅ **可追溯**：有明确的更新记录和决策依据
- ✅ **可实施**：开发/设计团队可直接基于文档工作

### 六类文档写作重点

#### 📂 00_README.md（章节目录）

**目标读者**：全体团队成员
**核心目的**：提供章节整体概览和快速导航

**写作重点**：

- 📊 **一页看全**：在一个页面内展示章节全貌
- 🎯 **快速定位**：清晰的子文档导航和状态标记
- 🔗 **关系梳理**：说明与其他章节的依赖关系
- 📈 **进度透明**：实时展示章节完成度

**必须包含**：

1. 章节概述（一句话说明、系统定位）
2. 文档导航表（5个子文档的链接、状态、负责人）
3. 核心概念速查（术语表、关键流程图）
4. 核心功能清单（已完成/进行中/待开发）
5. 技术难点摘要（主要挑战和解决方案）
6. 开发进度（各文档完成百分比）
7. 依赖关系（依赖哪些章节、被哪些依赖）
8. 待讨论问题清单
9. 更新记录

**示例结构**：

```markdown
# 01 · 宇宙地理系统

> **核心映射**：IP地址 → 星域/星系/星球  
> **优先级**：P0 | **状态**：✅ 完成

## 📖 章节概述
[一句话说明 + 系统定位]

## 📚 文档导航
| 文档 | 内容概要 | 状态 | 负责人 |
[5个子文档的导航表]

## 🔗 核心概念速查
[术语表 + 流程图]

## 🎯 核心功能清单
- [x] 功能1
- [ ] 功能2

## ⚠️ 技术难点
[主要挑战和解决方案]

## 📊 开发进度
[各文档完成百分比]

## 🔗 依赖关系
[依赖和被依赖的章节]

## 📝 待讨论问题
[未解决的问题列表]

## 📅 更新记录
[时间线]
```

**禁止出现**：过于详细的技术实现、完整的代码、详细的美术规格

#### 📖 01_世界观设定.md

**目标读者**：产品经理、策划、全体团队
**核心目的**：用故事化语言描述系统在宇宙中的意义和运作方式

**写作重点**：

- ✍️ **叙事优先**：用故事而非技术术语讲述
- 🎭 **用户视角**：从领主/臣民的角度描述体验
- 🌍 **世界融入**：展示系统如何融入整个宇宙世界观
- 📚 **术语定义**：明确世界观术语与现实技术的对应

**必须包含**：

1. 核心概念（一句话概述）
2. 背景故事（起源、演变、现状）
3. 运作机制（基本规则、关键流程）
4. 用户体验（第一次接触、日常使用、高级玩法）
5. 术语表（世界观术语 ↔ 现实对应）

**禁止出现**：技术细节、代码、API、数据结构

---

#### 🔗 02_现实映射.md

**目标读者**：产品经理、技术负责人
**核心目的**：建立技术概念与世界观表现的精确映射关系

**写作重点**：

- 🔄 **双向映射**：技术→世界观、世界观→技术都要说明
- 📊 **完整性**：覆盖所有核心技术点，不遗漏
- ⚖️ **合理性**：解释为什么这样映射（设计理由）
- 🚧 **边界清晰**：说明技术限制如何体现在世界观中

**必须包含**：

1. 技术→世界观转换表（如：IP地址→星球坐标）
2. 用户行为映射（如：添加好友→建立外交）
3. 技术限制的世界观解释（如：NAT→迷雾空间）
4. 异常情况处理（如：断网→虚空沉沦）
5. 与其他系统的接口映射

**示例格式**：

```markdown
| 技术概念 | 世界观表现 | 映射逻辑 | 用户感知 |
|---------|-----------|---------|---------|
| IP地址 | 星球坐标 | 数字→宇宙位置 | 看到星图定位 |
```

#### 🎨 03_资源与美术.md

**目标读者**：美术设计师、UI/UX设计师
**核心目的**：明确所有视觉、音效、交互资源需求

**写作重点**：

- 🎨 **视觉风格统一**：色调、风格、参考案例
- 📋 **清单化**：每个资源标注规格、数量、优先级
- 🔢 **可量化**：具体尺寸、格式、数量，便于排期
- 🎬 **动效描述**：关键交互的动画效果说明

**必须包含**：

1. 视觉风格定义（色调、风格、参考图）
2. UI资源清单（图标、界面、按钮等）
3. 场景资源清单（背景、特效、3D模型等）
4. 动效设计（过渡动画、交互反馈）
5. 音效需求（背景音乐、交互音效）
6. 资源优先级（P0/P1/P2）

**示例格式**：

```markdown
| 资源类型 | 规格 | 数量 | 优先级 | 状态 | 负责人 |
|---------|------|------|--------|------|--------|
| 星球图标 | 256×256 PNG | 10个 | P0 | 待设计 | - |
```

#### 💻 04_开发设计.md

**目标读者**：开发工程师、架构师
**核心目的**：提供完整的技术实现方案和开发指南

**写作重点**：

- 🏗️ **架构清晰**：数据结构、接口、状态机
- 📐 **可实现**：具体到可以直接编码的程度
- 🔗 **依赖明确**：列出技术依赖和前置条件
- ⚠️ **风险识别**：技术难点和解决方案

**必须包含**：

1. 数据结构定义（JSON Schema、数据库表）
2. API接口设计（端点、参数、返回值）
3. 核心算法/流程（流程图、伪代码）
4. 状态机设计（状态转换图）
5. 技术难点与解决方案
6. 依赖关系（依赖哪些系统、被哪些依赖）
7. 测试要点

**示例格式**：

```markdown
### API设计
**端点**：`POST /api/planet/register`
**参数**：
```json
{
  "planet_name": "string",
  "public_key": "string",
  "ip_address": "string"
}
```

**返回**：

```json
{
  "opid": "string",
  "sector": "string",
  "galaxy": "string"
}
```

```

#### ⚡ 05_事件与演化.md
**目标读者**：运营策划、游戏设计师  
**核心目的**：设计动态内容和系统成长机制

**写作重点**：
- 📅 **时间维度**：触发型、周期型、剧情型事件
- 📈 **成长路径**：系统如何随用户成长而变化
- 🎲 **随机性**：引入不确定性和惊喜感
- 📖 **故事性**：用事件推动世界观叙事

**必须包含**：
1. 触发型事件（条件、表现、影响）
2. 周期型事件（频率、内容、奖励）
3. 演化路径（新手→高级玩家的系统变化）
4. 随机事件池（好事/坏事/中性事件）
5. 故事线索（与世界观背景呼应的剧情）
6. 运营活动设计（节日、限时事件）

**示例格式**：
```markdown
| 事件名称 | 类型 | 触发条件 | 世界观表现 | 影响 |
|---------|------|---------|-----------|------|
| 坐标漂移 | 触发型 | IP变更 | 星球在宇宙中移动 | 需重新注册 |
| 宇宙大探险 | 周期型 | 每周一 | 星历局发布探索任务 | 奖励声誉值 |
```

## 🤝 协作流程

### 认领章节

1. 在项目看板上认领具体章节或文档
2. 创建对应的分支：`feature/章节名称`
3. 按照标准模板开始编写

### 文档编写

1. **第一轮**：先完成世界观设定（01文档）
2. **第二轮**：基于世界观完成现实映射（02文档）
3. **第三轮**：补充资源美术需求（03文档）
4. **第四轮**：详细技术开发设计（04文档）
5. **第五轮**：设计事件演化逻辑（05文档）

### Review 流程

1. 完成文档后提交 Pull Request
2. 至少2位团队成员进行 Code Review
3. 检查术语一致性、逻辑完整性
4. Review 通过后合并到主分支
5. 同步更新进度看板

### 版本管理

- 使用语义化版本号：`v主版本.次版本.修订版本`
- 每次重大更新创建 Release Tag
- 维护详细的 CHANGELOG.md
