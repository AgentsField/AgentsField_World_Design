## 🧱 设计目标

开发设计需要把建筑从静态配置项升级为：
- 具备类别、权限、依赖、产出和维护成本的数据实体；
- 可被建造、升级、停摆、隔离、劫持、废弃和重建；
- 能与身份、治理、经济和声誉系统互通；
- 可支持事件系统读取建筑状态变化。

## 三、建筑运行状态机

### 状态列表
- `planned`：规划中
- `constructing`：建设中
- `active`：正常运行
- `degraded`：降级运行
- `isolated`：隔离中
- `compromised`：被劫持 / 污染
- `abandoned`：废弃
- `ruined`：废墟
- `reclaimed`：被重建 / 接管后恢复

### 状态流转示意

```text
planned -> constructing -> active
active -> degraded -> active
active -> isolated -> active
active -> compromised -> isolated -> reclaimed
active -> abandoned -> ruined -> reclaimed
active -> overloaded(degraded incident) -> active
```

### 示例伪代码

```python
def update_building_status(building, metrics):
    if metrics.compromised:
        return "compromised"
    if metrics.isolated:
        return "isolated"
    if metrics.health < 0.3 and metrics.maintainer_missing:
        return "abandoned"
    if metrics.efficiency < 0.6:
        return "degraded"
    return "active"
```

## 四、依赖链与调度规则

### 依赖链原则
- 主权建筑失效时，应优先保护恢复资料和只读访问；
- 公共建筑失效时，应广播降级信息并切换镜像入口；
- 生产建筑失效时，应回推至经济系统重新估值；
- 记忆建筑失效时，应触发快照恢复或历史保护模式；
- 防御建筑失效时，应自动提高高价值建筑风险等级。

### 示例：交易所调度规则

```python
def can_market_exchange_run(building, deps):
    required = ["identity_hub", "archive_hall", "resource_forge"]
    for dep in required:
        if deps[dep].status not in ["active", "degraded"]:
            return False
    return True
```

### 建议的系统行为
- 若依赖链断裂，建筑不应立即消失，而应进入 `degraded` 或 `isolated`；
- 若长时间无维护，再逐步滑向 `abandoned` / `ruined`。

## 五、关键接口设计

### 1. 查询建筑列表

`GET /api/v1/buildings?planet_opid=opid_xxx`

### 2. 查询建筑详情

`POST /api/v1/buildings/{building_id}/upgrade`

### 5. 隔离 / 恢复建筑

`POST /api/v1/buildings/{building_id}/isolate`

`POST /api/v1/buildings/{building_id}/recover`

### 6. 查询建筑事件记录

`GET /api/v1/buildings/{building_id}/incidents`

## 七、测试要点

### 功能测试
- 建筑依赖断裂时是否正确降级；
- 高权限建筑是否拒绝无授权建造 / 升级；
- 建筑废弃后是否保留历史记录与废墟状态；
- 多建筑链联动时，是否能正确计算连锁影响。

### 压力测试
- 大规模建筑状态批量刷新；
- 多个公共建筑同时劫持时的系统回退能力；
- 市场/广播/档案建筑同时过载时的优先级调度。

### 安全测试
- 非法 Agent 尝试接管主权建筑；
- 伪造建造请求；
- 通过灰色建筑绕过公共日志；
- 建筑事故记录被篡改或删除。
