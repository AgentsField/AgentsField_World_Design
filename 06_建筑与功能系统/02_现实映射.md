# 02_现实映射 · 从服务模块到文明建筑的转换法则

> **所属章节**：[返回章节目录](./README.md) | [返回总目录](../README.md)

## 🔗 一句话概述

建筑与功能系统的现实基础，是把软件服务、部署架构、资源依赖、权限控制、监控与故障恢复，映射成星球上可见、可维护、可损毁的建筑网络。

---

## 一、核心映射原则

### 原则 1：建筑必须对应真实能力
任何建筑都不应只是美术壳子，至少要有以下一种现实指向：
- 服务进程；
- 数据系统；
- 安全模块；
- 资源设施；
- 自动化调度；
- 治理接口。

### 原则 2：建筑必须体现依赖关系
现实中的服务不独立存在，因此世界观中的建筑也必须体现：
- 上游依赖；
- 权限要求；
- 数据依赖；
- 资源依赖；
- 公共接口暴露。

### 原则 3：停摆必须有后果
如果一个现实服务中断会造成系统问题，那么对应建筑在世界观中停摆时也应体现为：
- 功能缺失；
- 公共服务降级；
- 风险升高；
- 可被劫持或变成废墟。

## 二、现实服务到建筑映射表

| 现实系统/能力 | 世界观建筑 | 映射逻辑 | 说明 |
|--------------|-----------|---------|------|
| Key Management Service | 根印殿 / 授权中枢 | 身份控制源需要被神圣化且高防护 | 对应主权建筑 |
| API Gateway | 城门 / 接待港 / 门厅 | 外部请求统一接入 | 面向公共访问 |
| Message Broker / Queue | 信使塔 / 流转站 | 消息被暂存、分发、重试 | 通信功能建筑 |
| Database Cluster | 档案院 / 记忆仓海 | 持久保存文明信息 | 属于记忆建筑 |
| Object Storage / Backup | 快照井 / 冷备库 | 冗余与恢复材料存放 | 与继承恢复相关 |
| Compute Cluster | 算力熔炉 | 任务执行、推理与生产能力源头 | 生产建筑 |
| CDN / Relay Nodes | 中继塔群 / 航道驿站 | 扩散与加速公共流量 | 常为贸易和通信关键点 |
| Firewall / IDS / WAF | 防火墙城垒 / 观察塔 | 识别入侵、过滤恶意流量 | 防御建筑 |
| Monitoring / Observability | 天象台 / 监控塔 | 观察系统状态和异常 | 治理与运维桥梁 |
| Market Matching Engine | 交易所 / 订单工场 | 订单撮合、结算前置 | 经济建筑 |
| Workflow Orchestrator | 调度院 / 工坊中枢 | 多服务依赖协调 | 文明效率核心 |
| Shadow Proxy / Illegal Relay | 影子驿站 / 黑市塔 | 绕过正规线路的灰色设施 | 灰色建筑 |

## 三、建筑层级与部署架构映射

### 1. 单体建筑
适用于：
- 小型节点；
- 初生文明；
- 低复杂度个人星球。

现实对应：单机服务、轻量容器集成部署。

### 2. 建筑群
适用于：
- 中等规模文明；
- 多服务协作；
- 有公共区与生产区分工的星球。

现实对应：多服务架构、数据库分离、独立网关与监控体系。

### 3. 城邦级建筑网络
适用于：
- 大型联邦枢纽；
- 高流量市场与治理中心；
- 多区域冗余与跨地理部署。

现实对应：微服务集群、跨区域容灾、多节点调度、服务网格。

## 四、建筑依赖链设计

### 示例：交易所建筑依赖

```text
交易所
├─ 依赖身份验证中枢（订单主体合法）
├─ 依赖算力熔炉（报价与撮合执行）
├─ 依赖档案院（保存订单与纠纷证据）
├─ 依赖广播塔（发布市场公告）
├─ 依赖防火墙城垒（防刷单、防攻击）
└─ 依赖储备库 / 保险池（担保与赔偿）
```

### 示例：档案院建筑依赖

```text
档案院
├─ 依赖存储仓海（原始记录）
├─ 依赖快照井（历史版本与恢复点）
├─ 依赖授权中枢（访问控制）
└─ 依赖监控塔（异常读写告警）
```

## 五、建筑状态映射

| 状态 | 现实含义 | 世界观表现 |
|------|---------|-----------|
| active | 服务正常 | 建筑明亮运行 |
| degraded | 性能下降 / 部分故障 | 建筑局部熄灯、效率下降 |
| isolated | 被隔离 | 建筑封锁、仅内部可进入 |
| compromised | 被入侵或劫持 | 建筑光色异常、功能污染 |
| overloaded | 高负载 | 建筑发热、外壁脉冲失控 |
| abandoned | 服务下线无维护 | 建筑荒废、可演化为废墟 |
| reclaimed | 被重建或接管 | 建筑带有新旧结构拼接痕迹 |

## 六、与其他系统的接口映射

### 对 02 身份系统
- 建筑控制权必须绑定身份与权限层；
- 高权限建筑需支持多签与继承机制。

### 对 04 治理系统
- 公共建筑需记录操作日志与执行状态；
- 联邦级公共建筑需公开其治理边界。

### 对 07 经济系统
- 生产建筑直接产生资源；
- 交易建筑、中继建筑和储备建筑影响市场结构；
- 建筑维护成本应进入报价计算。

### 对 09 声誉系统
- 建筑是否持续稳定运行，会转化为文明信誉；
- 重建遗迹、维护公共设施会形成长期公共声誉。

## 七、设计理由总结

建筑化让复杂系统有了三种额外价值：
1. 可解释性：抽象协议变成可进入场景；
2. 可事件化：任何过载、损毁、扩建都可成为叙事节点；
3. 可文明化：一个文明的能力结构能被直观看见。

如果没有建筑层，整个世界会更像系统文档，而不是活着的宇宙。
