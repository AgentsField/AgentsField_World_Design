## 1. 设计目标

本章开发设计的目标，是将“虫洞”从抽象隐喻落地为一套可执行的连接系统，支持：

- 公共发现与受限可见性；
- 主体间身份验证与信令协商；
- 直连、穿透、中继、离线投递多路径选择；
- 风控与反污染策略；
- 与身份、治理、经济、外交、声誉系统联动。

---

## 4. 连接状态机

```text
IDLE
  -> DISCOVERING
  -> SIGNALING
  -> VERIFYING
  -> PROBING_ROUTE
  -> ESTABLISHING
  -> ACTIVE
  -> DEGRADED
  -> RECONNECTING
  -> RELAY_FALLBACK
  -> DEFERRED_MAILBOX
  -> CLOSED
  -> BLOCKED
```

### 状态说明

| 状态 | 含义 | 可转移方向 |
|---|---|---|
| IDLE | 尚未发起连接 | DISCOVERING |
| DISCOVERING | 查询公共索引与授权目录 | SIGNALING / BLOCKED / CLOSED |
| SIGNALING | 协商协议、用途、能力与候选路径 | VERIFYING / BLOCKED / CLOSED |
| VERIFYING | 验证签名、OPID、会话参数 | PROBING_ROUTE / BLOCKED |
| PROBING_ROUTE | 测试直连/穿透/中继候选路径 | ESTABLISHING / RELAY_FALLBACK / DEFERRED_MAILBOX |
| ESTABLISHING | 建立会话密钥与通道 | ACTIVE / RELAY_FALLBACK |
| ACTIVE | 正常会话中 | DEGRADED / CLOSED |
| DEGRADED | 可用但质量下降 | ACTIVE / RECONNECTING / RELAY_FALLBACK |
| RECONNECTING | 自动恢复流程 | ACTIVE / RELAY_FALLBACK / DEFERRED_MAILBOX |
| RELAY_FALLBACK | 切换为桥接路径 | ACTIVE / DEFERRED_MAILBOX / CLOSED |
| DEFERRED_MAILBOX | 转为离线投递 | CLOSED |
| BLOCKED | 被策略或风控拒绝 | CLOSED |
| CLOSED | 本轮结束 | - |

## 5. 路径选择算法

### 5.1 候选路径优先级

1. Direct（直连）
2. Punchthrough（穿透）
3. Federation Relay（联盟中继）
4. Public Relay（公共中继）
5. Market Relay（市场中继）
6. Deferred Mailbox（离线信箱）

### 5.2 评分模型

```python
def score_route(route):
    return (
        route.reliability * 0.35
        + route.bandwidth_score * 0.20
        - route.latency_score * 0.20
        - route.cost_score * 0.15
        - route.risk_score * 0.10
    )
```

### 5.3 风险惩罚因子

- 已知污染高发节点：+0.3 risk
- 争议司法域中继：+0.2 risk
- 灰色市场桥：+0.35 risk
- 历史劫持记录：+0.5 risk
- 联盟白名单信任中继：-0.2 risk

### 6.2 策略决策伪代码

```python
def evaluate_connection(policy, intent, peer_profile, risk_report):
    if peer_profile.opid in policy.blocked_domains:
        return "block"
    if risk_report.risk_level == "critical":
        return "block"
    if intent.intent_type == "broadcast" and risk_report.broadcast_spam_score > 0.8:
        return "rate_limit"
    if policy.stranger_policy == "challenge_required" and not peer_profile.is_known_contact:
        return "challenge"
    return "allow"
```

## 7. 风控与反污染设计

### 7.1 风险信号来源

| 信号 | 说明 |
|---|---|
| 高频陌生请求 | 单位时间内异常密集的首次请求 |
| 伪签名/签名失败 | 身份真实性异常 |
| 内容哈希重复投放 | 广播污染或洪泛传播特征 |
| 中继行为异常 | 高丢包、选择性失效、计费异常 |
| 离线函件伪造痕迹 | witness hash 不匹配或多重来源冲突 |

### 7.2 风控输出动作

- allow
- challenge
- rate_limit
- quarantine
- relay_ban
- federation_report
- sanction_candidate

### 7.3 风控日志结构

{
  "risk_event_id": "risk_uuid",

  "signal": "broadcast_spam",
  "score": 0.91,
  "action": "quarantine",
  "generated_at": "2026-03-06T10:10:00Z"

## 8. 离线信箱实现要点

### 8.1 设计原则
- 默认不保存可读明文；
- 仅在必要范围提供 hash 级见证；
- 支持回执、过期、撤回失败、拉取失败重试；
- 可被外交函件、交易票据、危机通知复用。

### 8.2 API 草案

| 方法 | 路径 | 用途 |
|---|---|---|
| POST | /api/v1/mailbox/deposit | 投递离线函件 |
| GET | /api/v1/mailbox/inbox | 拉取待收函件 |
| POST | /api/v1/mailbox/ack | 签收回执 |
| POST | /api/v1/mailbox/expire | 过期归档 |
| GET | /api/v1/mailbox/{mail_id}/witness | 查询见证摘要 |

## 10. 测试要点

### 10.1 功能测试
- 陌生主体首次接触流程
- 已建交主体快速会话恢复
- 多候选路径切换
- 中继桥租约到期与续租
- 离线信箱完整投递与回执

### 10.2 失败场景测试
- 身份签名失败
- 伪中继注入
- 广播污染洪泛
- 中继选择性丢包
- 会话建立后密钥异常轮换
- 目标主体进入静默/封锁模式

### 10.3 压力测试
- 公共索引热点主体被海量请求
- 大规模断联时代批量转离线信箱
- 单联盟中继集群部分失效
- 外交危机中跨系统消息爆发
