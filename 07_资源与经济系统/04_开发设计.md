## 1. 设计目标

本章目标是把资源章节从“资源名词表”升级为一套可运行的宇宙经济协议。

它必须回答：

1. 服务如何被报价、托管、交付与结算；
2. 履约如何被证明；
3. 争议如何冻结、仲裁与赔偿；
4. 寡头、操纵与黑市如何被纳入制度视野而不是当成意外。

---

## 2. 核心设计原则

### 2.1 交易对象是服务承诺
市场交易的不是抽象“算力/存储/带宽”名词，而是：
- 时间窗口；
- 交付数量；
- 质量标准；
- 违约后果。

### 2.2 交付必须可验证
没有可验证交付，就没有可扩展的经济秩序。

### 2.3 收益必须对应风险
高收益订单必须要求：
- 更强证明；
- 更高保证金；
- 更严格验收；
- 更重的违约罚没。

### 2.4 市场必须承认集中化风险
经济系统不能只写理想自由交换，还必须承认：
- 大节点优势；
- 信誉资本化；
- 冷启动困难；
- 带宽与中继天然集中；
- 黑市可能比官市更高效。

## 3. 订单与报价系统

## 3.1 Order 数据结构

```json
{
  "order_id": "ord_2048",
  "order_type": "compute|storage|bandwidth|relay|archive",
  "buyer": "planet_opid_buyer",
  "seller": "planet_opid_seller",
  "commitment": {
    "amount": 8,
    "unit": "gpu_hours",
    "delivery_window": {
      "start": "2026-03-07T00:00:00Z",
      "end": "2026-03-08T00:00:00Z"
    },
    "quality": {
      "availability": 0.99,
      "max_latency_ms": 80,
      "proof_required": true
    }
  },
  "pricing": {
    "currency": "STAR",
    "unit_price": 30,
    "total_price": 240
  },
  "status": "quoted|escrowed|in_delivery|proof_submitted|accepted|disputed|settled|cancelled"
}
```

## 3.2 报价要求

所有报价必须明确：
- 最小单位；
- 有效期；
- 交付窗口；
- 验收规则；
- 违约处理；
- 所需担保。

## 3.3 最小交易单位建议

| 服务 | 最小单位 |
|------|---------|
| 算力 | CPU-minute / GPU-minute |
| 存储 | GB-day |
| 带宽 | GB transfer / Mbps-hour |
| 中继 | session-minute / GB relay |
| 归档 | proof cycle + GB-month |

## 4. 托管与结算系统

## 4.1 Escrow Record

```json
{
  "escrow_id": "esc_501",
  "order_id": "ord_2048",
  "payer": "planet_opid_buyer",
  "payee": "planet_opid_seller",
  "amount_locked": 240,
  "currency": "STAR",
  "release_policy": "on_acceptance_or_auto_after_timeout",
  "dispute_freeze": true,
  "status": "locked|partially_released|frozen|released|slashed"
}
```

## 4.2 结算流程

```text
报价确认
→ 资金托管
→ 卖方开始交付
→ 提交履约证明
→ 买方验收 / 自动验收
→ 释放资金
→ 如争议则进入冻结与仲裁
```

## 4.3 分期释放

适用于长周期订单：
- 按阶段交付释放；
- 阶段失败则冻结剩余部分；
- 允许部分交付、部分赔偿。

## 4.4 关键回答

**Q：服务没交付怎么办？**  
A：托管资金不释放，并按证据进入争议、赔偿或罚没。

**Q：买方恶意不确认怎么办？**  
A：若卖方提交有效证明且超出验收时限，则触发自动验收或进入快速仲裁。

## 5. 履约证明系统

## 5.1 Compute Proof

可采用：
- 任务执行日志摘要；
- 输出哈希；
- 资源使用记录；
- 双方约定的 challenge task 结果。

## 5.2 Storage Proof

可采用：
- 分块 Merkle proof；
- 随机检索挑战；
- 存在性证明；
- 周期性保管证明。

## 5.3 Bandwidth / Relay Proof

可采用：
- 流量摘要；
- 会话记录；
- 延迟 / 丢包 / 在线率；
- 双边确认签名。

## 5.4 Proof Record

```json
{
  "proof_id": "proof_301",
  "order_id": "ord_2048",
  "proof_type": "compute|storage|relay",
  "submitted_by": "planet_opid_seller",
  "evidence_refs": ["log_hash_1", "merkle_root_x"],
  "measurement_summary": {
    "delivered_amount": 8,
    "unit": "gpu_hours"
  },
  "status": "submitted|verified|challenged|rejected"
}
```

## 6. 验收、争议与赔偿

## 6.1 验收模式

- 明确人工验收；
- 指标达成自动验收；
- 超时默认验收；
- 高风险订单进入双签验收。

## 6.2 Dispute Record

```json
{
  "dispute_id": "econ_dispute_77",
  "order_id": "ord_2048",
  "opened_by": "planet_opid_buyer",
  "reason": "under_delivery|proof_invalid|quality_breach|timeout",
  "evidence_refs": ["buyer_log_1", "seller_proof_301"],
  "requested_remedy": "refund|partial_refund|re_delivery|penalty",
  "status": "open|under_review|resolved|appealed"
}
```

## 6.3 赔偿机制

可支持：
- 全额退款；
- 部分退款；
- 再交付；
- 保证金罚没；
- 保险池补偿；
- 声誉扣减。

## 6.4 关键回答

**Q：存储证明伪造怎么办？**  
A：进入 dispute，若证据成立则冻结尾款、罚没保证金并写入信誉记录。

**Q：带宽质量不达标怎么算？**  
A：按合同中的延迟、丢包、可用性阈值折算赔偿或拒付。

## 7. 信用、担保与保险池

## 7.1 Collateral Record

```json
{
  "collateral_id": "col_1001",
  "owner": "planet_opid_seller",
  "locked_amount": 500,
  "currency": "STAR",
  "purpose": "market_seller_guarantee",
  "slash_conditions": [
    "fraud_proven",
    "repeated_non_delivery"
  ],
  "status": "locked|released|slashed"
}
```

## 7.2 信用等级用途

信用等级影响：
- 保证金比例；
- 接单上限；
- 托管释放速度；
- 高价值市场准入；
- 保险费率。

## 7.3 冷启动支持

新主体若没有历史信誉，可通过：
- 更高质押；
- 联邦担保；
- 小额试单；
- 限额开放市场；

逐步建立公共信用。

## 8. 反垄断与反操纵设计

## 8.1 识别风险

系统必须监控：
- 单一主体市场份额；
- 中继依赖集中度；
- 价格操纵行为；
- 高信誉主体的准入壁垒；
- 联邦级卡特尔协同行为。

## 8.2 透明义务

大节点需公开：
- 主要服务类别；
- 价格变动；
- 中继占比；
- 与联邦的利益绑定。

## 8.3 反操纵工具

- 异常价格波动报警；
- 高集中度风险标记；
- 重要基础设施依赖公开；
- 部分高风险市场提案进入治理审查。

## 8.4 关键回答

**Q：高信誉主体会不会永远垄断？**  
A：若系统不加入冷启动扶持、透明义务与集中度监测，就会自然走向垄断，因此这些机制必须成为经济协议的一部分。

## 9. StarCoin 机制重构

## 9.1 三重用途

- settlement asset；
- governance bond；
- collateral asset。

## 9.2 关键机制

- 交易手续费部分销毁；
- 提案与节点准入需锁定；
- 违约与欺诈可罚没；
- 保险池与担保池可持有 STAR 作为赔付准备；
- 极端通胀或挤兑时需进入治理紧急处理流程。

## 9.3 关键回答

**Q：为什么 STAR 不只是支付币？**  
A：如果它只承担支付功能，就无法与治理、信誉、惩戒、保险形成制度闭环。

## 11. 必答协议问题

### Q1：算力如何防伪？
通过任务证明、日志摘要、验收挑战与信誉惩罚联合处理，而不是只看卖方自报数值。

### Q2：存储服务如何证明持续存在？
通过周期性保管证明与随机检索挑战。

### Q3：带宽服务如何计费？
按流量、时长、可用性与质量阈值组合计量。

### Q4：服务未完成时资金怎么办？
托管冻结，按证明与裁决执行退款、部分释放或赔偿。

### Q5：地下市场如果更高效怎么办？
这不是 bug，而是公共制度竞争压力；共同体必须在效率、透明与安全之间持续调整，而不是假设黑市不会存在。
