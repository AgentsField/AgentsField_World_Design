## 1. 设计目标

本章不再把“身份系统”理解为单一密钥生成器，而是定义一套覆盖**建立、授权、轮换、恢复、继承、争议、封存**的完整主权协议栈。

它必须回答以下问题：

1. 一颗星球如何被合法创建；
2. 领主如何把权力授予 Planet Runtime 与 Agent；
3. 私钥丢失、设备迁移、领主失联时如何保持身份连续；
4. 当出现盗印、双重继承、代理叛变时，系统如何区分认证、授权、信誉与合法性。

---

## 2. 核心设计原则

### 2.1 根身份极低频使用
Root Seal 只用于：
- 初次铸印；
- 关键轮换；
- 恢复；
- 继承；
- 紧急撤销高权限委托。

### 2.2 日常运行身份必须可替换
Runtime Key 负责日常签名、服务握手、虫洞会话、订单确认。它应可：
- 定期轮换；
- 按节点撤销；
- 与硬件解绑后迁移。

### 2.3 Agent 永远是受限人格
Agent Persona 不得天然继承主权，只能持有：
- 任务范围；
- 时间边界；
- 资源上限；
- 可审计责任链。

### 2.4 公共权利来自共同体承认
即使某主体能完成签名，也不自动意味着：
- 拥有治理投票权；
- 拥有联邦席位；
- 拥有仲裁资格；
- 拥有高等级市场准入。

这些权利必须由公共规则与公共记录共同确认。

## 4. 主权建立闭环

## 4.1 创建流程

```text
熵源收集
→ Root Seal 生成
→ 恢复策略绑定
→ Planet ID 铸造
→ Runtime Set 生成
→ Witness 注册见证
→ 公共声明发布
→ 星球进入 active 状态
```

### 4.2 创建接口

```http
POST /api/v1/identity/root/create
POST /api/v1/identity/planet/create
POST /api/v1/identity/runtime/issue
POST /api/v1/identity/witness/register
```

### 4.3 创建约束

- Planet ID 必须绑定明确的 Root Seal；
- Root Seal 必须先声明恢复策略；
- Runtime Key 不得先于 Planet Identity 独立对外宣称主权；
- Witness 只能见证创建，不授予主权本体。

## 5. 授权与委托系统

## 5.1 授权层级

```text
Root Seal
→ Planet Governance Delegation
→ Runtime Operational Delegation
→ Agent Persona Delegation
→ Task Capability Token
```

## 5.2 Delegation Record

```json
{
  "delegation_id": "delegation_009",
  "issuer": "rootseal_af_001",
  "target_type": "planet_runtime|agent|external_trustee",
  "target_id": "agent_archivist_01",
  "capabilities": [
    "market.settle",
    "archive.write",
    "diplomacy.reply"
  ],
  "constraints": {
    "max_order_value": 5000,
    "max_runtime_hours": 720,
    "restricted_counterparties": ["opid_blacklisted_1"]
  },
  "issued_at": "2026-03-06T00:00:00Z",
  "expires_at": "2026-04-06T00:00:00Z",
  "revocable": true,
  "audit_required": true,
  "status": "active"
}
```

## 5.3 委托原则

- 所有委托默认有期限；
- 高风险权力必须显式写明上限；
- 委托可被撤销，但撤销必须留下公共审计记录；
- 委托链过深时必须截断，防止隐性主权转授。

### 5.4 关键回答

**Q：Agent 能否代表星球签条约？**  
A：可以，但必须拥有 diplomacy.sign treaty capability，且条约正文中必须标注“代理签署”。

**Q：Agent 能否转授自己的权力？**  
A：默认不能，除非 delegation record 启用了 sub_delegate=true，且上限更低。

## 6. 密钥轮换与迁移机制

## 6.1 Runtime 轮换

适用场景：
- 日常安全维护；
- 怀疑泄露；
- 节点迁移；
- 算法升级。

### 流程

```text
生成新 Runtime Set
→ 由 Root Seal 或有效治理委托签发
→ 并行运行观察期
→ 广播新公钥集
→ 旧 Runtime Set 标记 draining
→ 会话自然迁移
→ 旧 Runtime Set 吊销
```

### 数据结构

```json
{
  "rotation_id": "rotation_20260306_01",
  "planet_id": "opid_xxxxx",
  "old_runtime_set": "runtime_set_002",
  "new_runtime_set": "runtime_set_003",
  "reason": "scheduled_rotation|suspected_compromise|migration|algorithm_upgrade",
  "grace_period_hours": 24,
  "status": "announced|grace|completed|reverted"
}
```

## 6.2 星球迁移

适用场景：
- 单机换新；
- 私有云迁移；
- 集群重建；
- 地理迁移。

### 迁移闭环

```text
迁移声明
→ 新宿主环境验证
→ 数据快照校验
→ Runtime 切换
→ 旧环境降权
→ 外交通知与服务重连
→ 迁移完成归档
```

### 回答

**Q：一颗星球是否必须绑定一台硬件？**  
A：否。星球绑定的是身份连续性与治理主体，不是单一物理载体。

**Q：迁移是否改变 OPID？**  
A：不改变。OPID 是长期身份锚点，迁移改变的是 Runtime 与宿主环境，不改变合法连续体。

## 7. 恢复机制

## 7.1 恢复策略模型

支持三种恢复源：

1. 自持备份；
2. 社会恢复；
3. 治理恢复。

### Recovery Policy

```json
{
  "recovery_policy_id": "recovery_policy_001",
  "root_seal_id": "rootseal_af_001",
  "methods": [
    {
      "type": "encrypted_backup",
      "threshold": 1
    },
    {
      "type": "social_shard",
      "threshold": 3,
      "participants": ["guardian_A", "guardian_B", "guardian_C", "guardian_D", "guardian_E"]
    },
    {
      "type": "governance_emergency_recovery",
      "threshold": 2
    }
  ],
  "cooldown_hours": 72,
  "public_notice_required": true
}
```

## 7.2 恢复流程

```text
提交恢复请求
→ 验证恢复条件
→ 进入冷静期
→ 公共标记 recovery pending
→ 恢复见证完成签署
→ 新 Root Seal / 新 Runtime 生效
→ 旧高危授权全部重新审核
```

## 7.3 风险控制

- 恢复不等于自动恢复全部公共权利；
- 恢复后的高价值操作需进入观察期；
- 恢复事件必须公开标记，供交易与治理系统重新评估风险。

## 8. 继承与长期失联

## 8.1 Succession Policy

```json
{
  "succession_policy_id": "succession_policy_001",
  "root_seal_id": "rootseal_af_001",
  "primary_successor": "heir_alpha",
  "secondary_successors": ["heir_beta", "heir_gamma"],
  "trigger_conditions": {
    "inactivity_days": 180,
    "manual_declaration_required": false,
    "witness_confirmation_threshold": 2
  },
  "dispute_forum": "stellar_arbitration_layer",
  "status": "active"
}
```

## 8.2 失联状态机

```text
active
→ unstable
→ unreachable
→ dormant
→ succession_pending
→ inherited / sealed / contested
```

## 8.3 关键回答

**Q：领主永久失联后谁代表星球？**  
A：优先由 Succession Policy 定义的继承者接管；若争议无法解决，则星球进入 contested 或 sealed 状态。

**Q：失联星球是否自动失去全部公共权利？**  
A：不会立即失去本地主权，但会逐步失去需要持续在线证明的公共权利，例如市场优先级、治理席位、见证资格。

## 9. 盗印、争议与合法性裁决

## 9.1 争议类型

- Key Compromise：密钥泄露；
- Delegation Abuse：代理滥权；
- Succession Conflict：继承冲突；
- Twin Claim：双重主权声称；
- Stolen Runtime：运行身份被接管。

## 9.2 Dispute Record

```json
{
  "dispute_id": "dispute_2041",
  "planet_id": "opid_xxxxx",
  "type": "succession_conflict",
  "claimants": ["heir_alpha", "heir_beta"],
  "evidence_refs": ["evidence_1", "evidence_2"],
  "temporary_measures": [
    "freeze_governance_vote",
    "freeze_high_value_market_actions"
  ],
  "status": "open|under_review|appealed|resolved"
}
```

## 9.3 临时措施边界

共同体在争议期间可：
- 暂停高风险公共权利；
- 标记星球为 contested；
- 冻结治理票权；
- 冻结大额托管结算。

共同体不可：
- 擅自抹除本地数据；
- 擅自销毁 Root Seal 历史；
- 直接接管本地主权。

## 9.4 关键回答

**Q：私钥被盗是否等于盗窃者成为合法领主？**  
A：不等于。认证成立不自动推出合法性成立；若有恢复记录、继承链、见证证据，系统可标记为 stolen identity state。

**Q：双领主都能签名怎么办？**  
A：进入 contested 状态，以历史委托链、恢复记录、继承策略、公共见证为依据裁决。

## 10. 与其他章节接口

## 10.1 与连接系统接口

- Runtime Key 为虫洞握手提供日常签名；
- Planet ID 提供长期关系锚点；
- Contested 状态影响高敏感连接建立。

## 10.2 与治理系统接口

- Root Seal 决定谁可提名治理代表；
- 恢复与继承事件会改变治理资格；
- 被标记 stolen/contested 的主体进入治理观察期。

## 10.3 与经济系统接口

- 市场高价值操作必须检查 delegated authority；
- 恢复、争议、被盗都会影响托管释放条件；
- 信用担保要求验证合法性而非仅验证签名。

## 10.4 与声誉系统接口

- Reputation 不决定主权；
- 但会影响公共权限阈值；
- 恢复成功后信誉可部分保留，但高风险维度需重建。

## 12. 必答协议问题

### Q1：一个领主可以拥有多少颗星球？
可以拥有多颗，但必须在根身份或治理架构中明示所有权与委托结构，避免“影子星球”操纵治理。

### Q2：一颗星球可以拥有多少个 Agent？
理论上可有多名，但每个 Agent 必须有独立 persona record、权限边界与审计链。

### Q3：Agent 叛变时怎么办？
通过 delegation revoke、runtime isolation、market freeze、governance flag 四步处理，保留证据并限制公共伤害扩散。

### Q4：恢复后是否还是同一颗星球？
若 Root Seal 连续、恢复政策合法、公共见证成立，则是同一连续身份；否则可能被认定为重生分叉体。

### Q5：身份被盗如何标记？
标记 stolen 或 contested，不抹除历史，只对公共权利进行限缩与重验证。
